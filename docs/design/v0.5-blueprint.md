# nxtLLM V0.5 蓝图设计文档

## 概述

V0.5 聚焦三个核心改进：消除 O(n) 驱逐瓶颈、实现碎片整理、落地 FlashInfer 流水线技术。目标是将 nxtLLM 从架构验证阶段推进到性能可用的推理引擎。

> **状态：✅ 全部完成** — 三个 Epic 均已实现、测试通过并推送到 master。

---

## Epic 1: 页面池哈希索引 ✅

### 1.1 问题

当前 `nxt_evict_lru_k()` 遍历整个 LRU-K 缓存（O(n)），对每个 entry 还需遍历空闲链表查找对应的 NxtPage*（再 O(n)）。在 >10K 页面的场景下，驱逐一次的代价不可接受。

### 1.2 方案

在 `NxtGlobalBufferPool` 中维护 `page_id → NxtPage*` 哈希表。

#### 哈希表设计

```c
/* page_pool.h 新增 */

typedef struct NxtPageEntry {
    uint32_t          page_id;
    NxtPage          *page;
    struct NxtPageEntry *next;    /* 链式冲突解决 */
} NxtPageEntry;

typedef struct {
    NxtPageEntry **buckets;
    uint32_t       bucket_count;  /* 2 的幂次，方便位运算取模 */
    uint32_t       entry_count;
} NxtPageHash;

/* 嵌入 NxtGlobalBufferPool */
typedef struct {
    /* ... 现有字段 ... */
    NxtPageHash page_hash;        /* NEW: page_id → NxtPage* */
} NxtGlobalBufferPool;
```

#### API

```c
/* 内部函数，不对外暴露 */
static void     page_hash_init(NxtPageHash *hash, uint32_t capacity);
static void     page_hash_destroy(NxtPageHash *hash);
static void     page_hash_insert(NxtPageHash *hash, NxtPage *page);
static void     page_hash_remove(NxtPageHash *hash, uint32_t page_id);
static NxtPage *page_hash_lookup(const NxtPageHash *hash, uint32_t page_id);
```

#### 哈希函数

FNV-1a 或简单的乘法哈希，取模用位运算（`page_id & (bucket_count - 1)`）。

#### 集成点

| 位置 | 操作 |
|------|------|
| `nxt_pool_init` | 初始化哈希表，插入所有分配的页面 |
| `nxt_page_alloc` | 从哈希表标记页面为 in-use（或移除） |
| `nxt_page_free` | 重新插入哈希表 |
| `nxt_evict_lru_k` | 用 `page_hash_lookup(page_id)` 替代链表遍历 |

#### 性能预期

| 操作 | Before | After |
|------|--------|-------|
| 查找 page by id | O(n) 遍历链表 | O(1) 哈希查找 |
| 驱逐一次 | O(n²) | O(k)，k = 候选数 |
| 内存开销 | 0 | ~24 bytes/page |

#### 测试计划

- 哈希表 CRUD 正确性
- 冲突处理验证
- 大规模（100K pages）驱逐性能对比
- 并发安全边界（单线程场景，但验证无竞争）

## Epic 2: 后台碎片整理 ✅

### 2.1 问题

长时间运行后，频繁 alloc/free 导致同 tier 内页面物理地址不连续，空闲页分散在各处。`nxt_defrag_background()` 目前仅递增计数器。

### 2.2 方案

#### 算法

```
nxt_defrag_background(pool):
  for each tier (GPU → CPU → SSD):
    1. 收集该 tier 所有 in-use 页面
    2. 按 data 指针物理地址排序
    3. 从低地址开始紧凑排列（memmove）
    4. 重建该 tier 的空闲链表（剩余空间连续）
    5. 更新 tier_used / free_counts
```

#### 数据结构

```c
/* memory_manager.c 内部 */

typedef struct {
    NxtPage **sorted_pages;    /* 按物理地址排序的页面数组 */
    int32_t   count;
} NxtTierPages;
```

#### 关键步骤

**Step 1: 收集已分配页面**
- 遍历 LRU-K 缓存，通过哈希表查找 NxtPage*
- 筛选 `page->data != NULL && page->ref_count > 0` 的页面

**Step 2: 按物理地址排序**
- 比较函数：`page_a->data < page_b->data`
- 使用 qsort，O(n log n)

**Step 3: 紧凑排列**
- 从最低地址开始，依次 memmove 页面数据
- 跳过已在正确位置的页面（优化）
- 更新 `page->data` 指针

**Step 4: 重建空闲链表**
- 收集所有未使用的 NxtPage 结构体
- 将其 data 设为 NULL（释放旧缓冲区）
- 追加到空闲链表尾部

**Step 5 (可选): 热页提升**
- 利用 LRU-K hit_count 排序
- 将高频访问页面迁移到更快 tier（GPU ← CPU）

#### 碎片率指标

```c
float nxt_fragmentation_ratio(const NxtGlobalBufferPool *pool) {
    /* 空闲碎片数 / 总空闲页面数 */
    /* 1.0 = 完全碎片化, 0.0 = 完全连续 */
}
```

#### 线程安全

- 需暂停所有 alloc/free 操作（或持全局锁）
- 建议在推理间隙（batch 间隙）执行
- 未来可改为增量式（每次整理一个 tier）

#### 测试计划

- 人工构造碎片场景 → 整理 → 验证连续性
- 碎片率计算正确性
- 整理后页面数据完整性（内容不被破坏）
- 跨 tier 不误操作

## Epic 3: Paged Attention V2 流水线 ✅

### 3.1 问题

当前 `paged_attention_v1_kernel` 使用传统加载模式，每步等待全局内存读取完成，SM 利用率低。FlashInfer 分析显示 cp.async 流水线可提升 30-50% decode 吞吐。

### 3.2 方案

创建 `operators/page_attention_v2.cu`，实现 FlashInfer 风格的 batch decode 内核。

#### 内核签名

```c
/* 与 v1 相同签名，新增 v2 入口 */
void nxt_paged_attention_v2(
    void* out, const void* query,
    const void* key_cache, const void* value_cache,
    const int* block_tables, const int* seq_lens,
    int num_seqs, int num_heads, int head_size,
    int num_kv_heads, float scale,
    int max_num_blocks_per_seq,
    int block_size, int dtype_size,
    int kv_block_stride, int kv_head_stride,
    nxt_stream_t stream);
```

#### 模板参数

```cpp
template <
    typename scalar_t,    // half / float
    typename cache_t,     // half / float
    int HEAD_SIZE,         // 64, 80, 96, 112, 128, 192, 256
    int BLOCK_SIZE,        // 16
    int NUM_THREADS,       // 128
    int GROUP_SIZE,        // 1, 2, 4, 8 (GQA)
    int VEC_SIZE           // 16 / sizeof(scalar_t)
>
__global__ void paged_attention_v2_kernel(...)
```

#### cp.async 流水线

```
Stage 0: issue cp.async K[0..tile], V[0..tile]
Stage 1: issue cp.async K[tile..2tile], V[tile..2tile]
──────────────────────────────────────────
Iter loop:
  1. cp.async.commit_group
  2. cp.async.wait_group<N-1>  → K ready in smem
  3. compute Q·K (fused scale + softmax update)
  4. issue cp.async K[next_tile]
  5. cp.async.wait_group<N-1>  → V ready in smem
  6. update O += W × V (online softmax output)
  7. issue cp.async V[next_tile]
```

#### GQA 线程布局

```
grid:   (num_seqs, num_kv_heads)
block:  (bdx=32, bdy=GROUP_SIZE, bdz=1)

每个 block 处理一个 (seq, kv_head) 对
bdy 维度处理同一个 kv_head 下的多个 query head
warp 内归约 softmax state (m, d, o)
```

#### 共享内存布局

```
smem per block:
  K_tile[bdy][tile_size][head_dim]     // KV page tile
  V_tile[bdy][tile_size][head_dim]
  m_smem[bdy]                          // softmax max
  d_smem[bdy]                          // softmax sum
  o_smem[bdy][head_dim]               // partial output
```

#### SM 架构检测与回退

```c
/* host launcher */
int sm_version;
cudaDeviceGetAttribute(&sm_version, cudaDevAttrComputeCapabilityMajor, device);

if (sm_version >= 8) {
    /* launch v2 kernel (SM80+ only) */
    nxt_paged_attention_v2<<<grid, block, smem_size, stream>>>(...);
} else {
    /* fallback to v1 */
    nxt_paged_attention<<<grid, block, smem_size, stream>>>(...);
}
```

#### CMake 集成

- v2 内核编译仅在 `USE_CUDA=ON` 时启用
- SM 架构限制：`CUDA_ARCHITECTURES "80;86;89;90"`
- 新增测试目标 `test_attention_v2` 对比 v1/v2 数值一致性和性能

#### 性能预期

| 场景 | v1 (baseline) | v2 (projected) |
|------|---------------|-----------------|
| Batch=1, ctx=1024, fp16 | 1.0× | 1.3-1.5× |
| Batch=16, ctx=1024, fp16 | 1.0× | 1.5-2.0× |
| Batch=64, ctx=4096, fp16 | 1.0× | 2.0-2.5× |

#### 测试计划

- 数值精度：v1 vs v2 输出误差 < 1e-3（fp16）
- 多种 HEAD_SIZE (64/128/256) 正确性
- GQA group_size (1/2/4/8) 正确性
- SM75 回退到 v1
- 性能基准：各配置的 kernel time 对比
